陈冠宇、杨过、糖醋排骨 等人赞同
无论是我之前所在的腾讯AlloyTeam团队， 还是现在所在的小米开发者生态团队， 都早已将Sass和Compass用在实际的项目开发中了。

在这里更想提一些Sass和Compass实战开发中遇到的坑：

1. 过多的使用嵌套
嵌套可以说是Sass中最简单的功能， 但正因为简单往往被大家滥用。看下边的代码：
body.mypage {
  div.profile {
    height: 40px;

    h2.name {
      font-size: 16px;

      p {
        color: red;
      }
    }
  }
}
最后最内层的p元素的选择器经过编译之后就变成了：
body.mypage div.profile h2.name p
我们都知道CSS是从右往左解析的， 这就意味着浏览器要首先找到所有的p元素，然后再看这些p元素中谁的父级元素是h2.name 然后再依次往上， 这样做无疑将大大延长排版引擎的处理时间。当然了，最近的浏览器在CSS解析方面都做了优化，所以这里的耗时相比于过去可以忽略不计了。
但还有第二个问题， 就是这种写法的可重用性。 如果要将这组样式应用到 .mypage 之外的地方， 尽管样式几乎完全一致，但是由于选择器不同，只有两种做法，要么为重用的地方也加上.mypage这样的类名， 要么把同样的样式完完全全的拷贝一遍，无论选择哪种，对于可扩展性来说都是一个噩耗。 不论是 OOCSS 还是 SMACSS 都强调按照模块来组织CSS， 以便提高其可重用性。所以限定在某个页面内的这种嵌套过深的选择器应该极力避免，此外根据BEM，我们能基于严格的命名规则用CSS类名来表现HTML结构，因此能避免使用嵌套就应该避免使用嵌套，而应该选择特定的class name的这种方式。一般来说嵌套的层级不应该超过三级， 如果超过三级， 我们就应该对模块的粒度进行重新的思考了。

2. 过多的使用 mixin
使用mixin可以让代码结构变得更加清晰，可维护性更高，但是有一个问题，mixin的代码在最终编译的时候会直接出现在每一个include调用的地方。这就意味着调用的地方越多，出现的次数就越多，CSS文件的最终大小就越大， 加载耗时就越长。还是直接上代码：
@mixin clearfix {
  &:before,
  &:after {
    content: " ";
    display: table;
  }
  &:after {
    clear: both;
  }
}

.foo {@include clearfix;}
.bar {@include clearfix;}
.baz {@include clearfix;}
上述的代码基于这样一个前提： 某个页面的页面布局使用float这种方式，并且需要使用clearfix来清除浮动，如果每一个需要清除浮动的地方都需要调用一次clearfix mixin, 那则意味着代码量要成倍增加。 那这时候更优的解决方案应该是啥样的呢？可以使用形如
<div class="float-box clearfix">
这样的结构， 额外增加一个clearfix类， 然后对clearfix类的修饰，调用 clearfix mixin, 这样基本上就不会出现冗余了。

所以我的做法是，一般不调用不需要传参的mixin，因为不使用参数的mixin，就意味着编译后的代码格式是固定不变的， 基本上都可以用上述多重样式类的方案解决。

3. 过多的使用extend
还是拿上边的clearfix举例子， 我们把它改写成extend的形式：
%clearfix {
  &:before,
  &:after {
    content: " ";
    display: table;
  }
  &:after {
    clear: both;
  }
}

.foo {@extend %clearfix;}
.bar {@extend %clearfix;}
.baz {@extend %clearfix;}
虽然规则没有在生成的CSS里边重复， 却产生了很多 .foo:before, .bar:before 这样的选择器， 还是没有我们上边提的多重样式类的方案来的优雅。

当然并不是说extend不好， 这时候就一定要具体问题，具体分析，比如说下边的这种场景应用：
%button {
  padding: 10px 30px;
}

.yellow-button {
  @extend %button;
  background-color: yellow;
}

.blue-button {
  @extend %button;
  background-color: blue;
}

针对按钮这种有一定程度限制的模块使用样式继承，并不会产生太多冗余，同时还保持了结构的清晰。另外这样做的优势也在于可以在不改变html文件元素的样式类的情况下仅通过修改css修改其样式。

说了以上三点其实就是想说明： Sass并非万能。 Sass虽然方便， 不要过分依赖，贸然使用某些Sass的功能。 用的时候一定要考虑： 为什么要用， 用了带来的好处是啥， 是可维护性？是为了精简代码量？是为了减少人工行为？ 总之一定要想清楚。

最后介绍一个小技巧： compass stats 命令可以统计项目的各项sass指标， 比如说mixin的定义数以及被调用的次数， 选择器数量， 样式属性数量， 文件大小等， 时常review这些指标并相应优化一定错不了。 显示全部
